# Документ ADR-001: System Style

**Статус:** Accepted

**Дата:** 07.04.2026

**Проєкт:** EsportHub

**Контекст:**
Система має забезпечувати швидкий рендер інтерактивної мапи світу (NFR-02: < 2.5 сек) та миттєве оновлення результатів кіберспортивних матчів (NFR-01: 3-5 сек). Команда розробки складається зі студентів, досвід налаштування розподілених інфраструктур (Kubernetes, Docker Swarm) обмежений. Технічні обмеження: нульовий бюджет на хостинг (необхідність використання Free Tier сервісів), жорсткі терміни до захисту MVP. Система повинна стабільно тримати навантаження до 50 одночасних підключень (NFR-03).

**Рішення:**
Ми обираємо **Модульну Монолітну архітектуру (Modular Monolith)**. 
Система розгортатиметься як єдиний процес (один сервер/контейнер), але внутрішня структура коду буде суворо розділена на логічні модулі з низькою зв'язністю (Low Coupling): `Auth`, `Map`, `Wiki`, `LiveMatches`.

**Обґрунтування (Rationale):**
Монолітна архітектура дозволяє уникнути критичного інфраструктурного Overhead-у на старті проєкту. Для команди пріоритетом є реалізація складного функціоналу (інтеграція з Liquipedia, WebSocket для live-рахунків), а не витрачання часу на налаштування Service Discovery та міжсервісної мережі.

* **Аргумент 1 "За" (Мінімізація Latency для Live-оновлень):** Відповідно до NFR-01, оновлення рахунку має відбуватися за 3-5 секунд. У моноліті модуль отримання даних та модуль WebSocket-розсилки знаходяться в одній пам'яті сервера. Це гарантує мінімальну затримку (Latency), якої складно досягти в мікросервісах через мережеві виклики.
* **Аргумент 2 "За" (Швидкість розробки та Time-to-Market):** Оскільки команда невелика і має обмежений час, єдиний репозиторій та запуск усього бекенду однією командою дозволяють швидко тестувати взаємодію між Вікіпедією та Мапою на локальних машинах без підняття складної інфраструктури.
* **Аргумент 3 "За" (Відповідність обмеженням хостингу):** Моноліт ідеально вписується у Free Tier обмеження хмарних провайдерів (напр. Render/Railway). Крім того, налаштування HTTPS (NFR-05) для єдиного домену моноліту значно простіше і безкоштовне.

**Перспектива:** Завдяки модульності коду, ми забезпечуємо гнучкість на майбутнє. Якщо навантаження зросте вище 50 користувачів (NFR-03) і модуль `LiveMatches` почне споживати забагато ресурсів, ми зможемо виділити його в окремий мікросервіс без необхідності переписувати бізнес-логіку `Wiki` або `Map`.

**Наслідки (Consequences):**
* **Позитивні:** Легке локальне налагодження (debugging); висока швидкість обміну даними між модулями (виклики функцій у пам'яті); спрощена робота з реляційною БД (PostgreSQL) без ризику втрати узгодженості даних (Data Consistency).
* **Негативні:** Single Point of Failure (єдина точка відмови) — якщо критична помилка в модулі `LiveMatches` (наприклад, через переповнення пам'яті при WebSockets) "покладе" сервер, перестане працювати вся система, включно з мапою та авторизацією. Крім того, неможливо масштабувати лише один модуль — доведеться дублювати весь моноліт.

**Альтернативи:**
* **Microservices (Мікросервіси):** Розглядалися, але були відхилені. Налаштування міжсервісної взаємодії (Message Brokers або gRPC) створить величезний інфраструктурний Overhead, що забере значну частину часу від розробки MVP. Також це збільшує ризики порушення NFR-01 через непередбачувану мережеву затримку між сервісами.
* **Serverless (Хмарні функції):** Відхилено через складнощі з підтриманням постійних WebSocket-з'єднань, що є критичним для забезпечення Live-оновлень платформи.
