# Data Flow & Interaction Specification-001

**Статус:** Accepted  
**Дата:** 07.04.2026  
**Проєкт:** EsportHub  

**Description:**

### 1. Високорівнева схема (High-Level Architecture)
Система базується на трирівневій моделі взаємодії:
* **Frontend Layer (Client):** Побудований на **React**. Відповідає за UI (зокрема швидкий рендер інтерактивної мапи), обробку WebSocket-з'єднань на боці клієнта та зберігання JWT-токена.
* **Backend Layer (API):** Побудований на **Node.js** (згідно з обраною архітектурою Модульного моноліту). Виконує роль REST API та WebSocket-сервера, обробляє бізнес-логіку (агрегація даних з Liquipedia) та керує авторизацією.
* **Data Layer (Storage):** База даних **PostgreSQL**. Забезпечує структуроване збереження профілів гравців, турнірів та архіву матчів.

### 2. Сценарії взаємодії (Data Flow)
Опишемо шлях даних для 3-х ключових функцій системи:

**Сценарій А: Завантаження Інтерактивної Мапи (згідно з NFR-02)**
1. **User:** Відкриває сторінку з глобальною мапою.
2. **Frontend:** Відправляє GET-запит на `/api/v1/map/players` для отримання геоданих.
3. **Backend (API Layer):** Передає запит у `MapService`.
4. **Backend (Service Layer):** Робить вибірку локацій активних гравців з БД та, за потреби, збагачує їх даними з REST Countries API. Формує оптимізований список гео-міток.
5. **Backend:** Повертає статус `200 OK` та масив об'єктів `MapMarkerDTO`.
6. **Frontend:** Отримує дані та рендерить маркери на мапі (процес займає < 2.5 сек).

**Сценарій Б: Live-оновлення результатів матчу (згідно з NFR-01)**
1. **Backend:** Модуль `LiveMatchService` фіксує зміну рахунку (через парсинг Liquipedia або зовнішній вебхук).
2. **Backend (WebSocket Layer):** Сервер формує повідомлення `LiveScoreDTO` (ID матчу, новий рахунок) і миттєво розсилає його через подію `score_update` всім підключеним клієнтам.
3. **Frontend:** WebSocket-клієнт отримує подію у фоновому режимі.
4. **Frontend:** Оновлює стан (State) компонента `LiveScoreBoard` без перезавантаження сторінки.
5. **User:** Бачить зміну рахунку на екрані із затримкою не більше 3-5 секунд.

**Сценарій В: Авторизація користувача (Discord / Google)**
1. **User:** Натискає кнопку «Увійти через Discord/Google».
2. **Frontend:** Перенаправляє користувача на сторінку OAuth провайдера.
3. **Провайдер:** Після успішного входу повертає код авторизації на наш Backend.
4. **Backend:** Отримує дані профілю, знаходить або створює нового юзера в PostgreSQL.
5. **Backend:** Генерує **JWT-токен**, що містить `UserId` та базові права доступу.
6. **Backend:** Перенаправляє на Frontend із токеном (в URL або через HttpOnly Cookie).
7. **Frontend:** Зберігає токен та використовує його для наступних закритих запитів (Bearer Token). Сервер не запам'ятовує сесію (Stateless).

### 3. Механізм передачі даних та Statelessness
* **Використання DTO (Data Transfer Objects):** Ми не передаємо сутності бази даних (Entities) безпосередньо на фронтенд. Використовуються спеціальні класи (наприклад, `PlayerProfileDTO` або `MapMarkerDTO`), щоб приховати технічні поля БД, паролі чи зайву метаінформацію, передаючи лише ті дані, які необхідні для швидкого рендеру. Це зменшує обсяг трафіку та підвищує безпеку.
* **Принцип Statelessness:** Наш Backend не зберігає стан користувача (Session ID) у своїй пам'яті. Кожен захищений запит від клієнта містить у заголовку JWT-токен. Це робить наш Модульний моноліт готовим до масштабування: якщо в майбутньому навантаження зросте, ми зможемо запустити другу копію Node.js сервера, і вона зможе без проблем обробити запит користувача, оскільки вся необхідна інформація для авторизації вже зашита в самому токені.
